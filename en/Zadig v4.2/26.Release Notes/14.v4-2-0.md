Zadig v4.2.0 was officially released on February 12, 2026.

## Feature List

**Workflow and Release Plan Enhancements**
- Workflows support system and project shared variables
- Workflows support restart service tasks
- Workflows support APISIX change tasks
- Workflow Apollo tasks support configuration search
- Workflow DMS data change work order tasks support selecting work orders for execution
- Workflows support tracing Feishu work item information
- Workflow global variables support being set as sensitive information
- Workflows support Lark (Feishu International) application notifications
- Workflow execution conditions support getting current task variables
- Workflow host service deployment tasks support fixed values for environments
- Workflow custom tasks support dynamic variables
- Workflow deployment tasks add deployment logs, showing resource change processes
- Workflow "Reference Selected Task Code Information" supports directly using pre-task code information
- Tests support test case pass rate configuration
- Host builds support using templates
- Release plan release items support specifying executors
- Release plans allow release managers to retry workflows
- Feishu approval link method optimization, improving stability
- Release plan permissions refined and split, supporting separate control of editing metadata, editing release items, and editing approvals

**Environment Management and Visualization Improvements**
- Environment service debugging supports specifying containers
- Environment service list displays Ingress Path information
- Helm service updates support viewing K8s manifest differences
- Environment service list images changed to current Workload images

**Openness and System Integration Optimization**
- Release plan OpenAPI supports passing global variables
- Execute workflow OpenAPI supports custom notification users
- Execute workflow OpenAPI supports multi-service service tests/code scanning/general tasks
- New create and delete sub-environment OpenAPI
- New collaboration mode create and delete OpenAPI
- New Helm scenario OpenAPI, including creating services using templates, creating environments, adding services to environments, etc.
- Support configuring system access address for internal network system callbacks
- Support project and system-level shared variable management

**Bug Fixes and Experience Optimization**
- Layout optimization when there are many project groups
- Workflow logs support quick copy
- Display reference information when deleting image repositories
- Add/update helm service list supports batch deletion
- Clone workflow build tasks clean up merged MRs
- Service real-time logs support filtering by line count and time
- When executing workflow tasks, if variables exist, they are displayed by default
- Workflow default approval information project name field changed to name instead of identifier
- K8s YAML services/Dockerfile/build/code scanning/release plan templates support copying
- Optimize real-time log search performance
- Optimize workflow execution version list selection performance
- Optimize workflow deployment task optional service list
- Optimize create version OpenAPI selecting from environment requiring source image issue
- Fix AWS cluster IRSA not working issue
- Fix SSH code source integration not supporting subpath issue
- Fix issue where tasks are still running when canceling workflow tasks
- Fix issue where sensitive information variables are not masked when building images
- Fix Statefulset service deployment status detection inaccuracy issue
- Fix issue where commit link is missing address in workflow task start event notifications
- Fix issue where branches are not displayed when cloned workflow build task branches exceed 200
- Fix issue where Commit is missing in image information generated when manually executing build stages
- Fix issue where service global variables (single-select type) optional range changes are not effective in environments
- Other minor bug fixes and frontend experience optimizations

## Business Change Statement

Release plan permissions are refined and split, supporting separate control of editing metadata, editing release items, and editing approvals.

## Version Upgrade Process

::: warning
If MongoDB is deployed in Replica Set mode, please confirm that system databases such as admin/local/config/test are not used as Zadig's business database. If they are used, please manually migrate the data first.

If the current system version < v4.1.0, please upgrade to v4.1.0 first. For specific upgrade steps, please refer to [v4.1.0 Upgrade Method](/en/Zadig%20v4.2/release-notes/v4.1.0/#version-upgrade-process), and then upgrade to v4.2.0 according to the following method.
:::

Please execute the corresponding upgrade process according to different installation methods:

- [Script Command Installation](/en/Zadig%20v4.2/install/helm-deploy/#upgrade)
- [Helm Command Installation](/en/Zadig%20v4.2/install/helm-deploy/#upgrade)
- [Official Channel Installation](/en/Zadig%20v4.2/stable/install/#upgrade)
